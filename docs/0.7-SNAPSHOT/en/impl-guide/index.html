The purpose of this document is to provide guidelines for proceeding efficiently and consistently with implementation work in this project.

This book is aimed at developers participating in this project. We aim to efficiently develop high-quality systems by developers complying with the processes shown in this document and the standards shown in the architecture specifications.

How to continue reading the guide

We recommend that you read this book in the following order. This order is designed so that developers can check the main points of this document and architecture specifications while referring to the actual working

application and source code at hand.

  1. Building an environment
    Follow the development environment construction procedure and build an execution/development

    environment on the PC used for development.

  2. Check the application
    Run the application in the built environment. Manipulate the application against the use cases in the architecture specification

    and check the implemented functionality.

  3. Review the application architecture

    1. logical configuration
      Refer to the application configuration and component configuration in the architecture specification to confirm the components that make up the application and

      their responsibilities.

    2. physical implementation
      Review the implementation process and the source code for each component described there

      to see how the component was implemented.

Implementation process

The implementation process guides how to implement the source code, where to place its files, and the order in which they are implemented.

The implementation method explains the role of each line of source code using a reference implementation as an example, and developers can implement source code that conforms to architecture specifications by referring to this and implementing their own scope of responsibility.

The implementation order is structured so that the implementation results can be immediately run and checked as an application.

Implementation processes are categorized by process, and developers can refer to processes in the same classification as their area of responsibility.

Registration screen

Here, I will explain the implementation process of the registration screen. The explanation takes the ticket registration screen of the reference implementation as an example.

Figure 1. Example of registration screen Ticket registration screen

Items for entering registration details and a button to execute registration are placed on the registration screen. Also, there are the following two processes on the registration screen

.

  1. Initialization process when the screen transitions

  2. Registration process for data entered on the screen

The implementation for each process is explained below.

Initialization process

The sequence of initialization processing on the registration screen is as follows.

.
Table 3. Processing sequence when the screen is initially displayed

  1. The user is transferred to the registration screen.

  2. PageLoader's load function is called.

  3. PageLoader creates an instance of Model that stores screen input values. Set initial values for screen items in instance properties

    4. .

  4. PageLoader calls a web API to obtain code values, etc. used on the screen. The obtained results are returned to page as a model.

  5. The Controller invokes the Service.

  6. Service calls the Repository.

  7. The repository performs SELECT against the DB.

  8. The controller converts the entity obtained from the service to DTO.

    9. Page constructs the HTML for

  9. the screen based on the model obtained from PageLoader. It also defines validation for input items

Hereafter, we will implement each element of the processing sequence described above in the following order.

  1. Frontend: Implementing appearance and validation

  2. DB: Implementing a migration script to create tables

  3. Backend: Web API implementation

  4. Frontend: Generating and Embedding Web API Clients

service

Create a Java service file under the domain package.

svqk-back/src/main/java/dev/aulait/svqk/domain/issue/IssueService.java
package dev.aulait.svqk.domain.issue;

import java.util.List;

import jakarta.enterprise.context.ApplicationScoped;
import lombok.RequiredArgsConstructor;

@ApplicationScoped (1)
@RequiredArgsConstructor (2)
public class IssueStatusService {

  private final IssueStatusRepository statusRepository; (3)

  public List<IssueStatusEntity> findAll() {
    return statusRepository.findAll(); (4)
  }

}

  1. Set @ApplicationScoped to the Service class.

  2. Set @RequiredArgsConstructor to the Service class. This makes it possible to use a constructor with a private final field as an argument without implementing it.

  3. The repository used by the service is defined as a private final field. This field is injected into

    the Repository instance when Quarkus is started.

  4. Retrieve an Entity from the Repository.

Controllers

Create a controller Java file under the interface package.

package dev.aulait.svqk.interfaces.issue;

import java.util.List;

import dev.aulait.svqk.arch.util.BeanUtils;
import dev.aulait.svqk.arch.web.ApiPath;
import dev.aulait.svqk.domain.issue.IssueStatusService;
import jakarta.ws.rs.GET;
import jakarta.ws.rs.Path;
import lombok.RequiredArgsConstructor;

@Path(IssueStatusController.ISSUE_STATUSES_PATH) (1)
@RequiredArgsConstructor (2)
public class IssueStatusController {

  private final IssueStatusService service; (3)

  static final String ISSUE_STATUSES_PATH = ApiPath.ROOT + "/issue-statuses";

  @GET (4)
  public List<IssueStatusDto> get() {
    return BeanUtils.mapAll(service.findAll(), IssueStatusDto.class); (5)
  }
}

  1. Set @Path to the Controller class. This allows this Controller class to be recognized as an API endpoint

    .

  2. Set @RequiredArgsConstructor to the Controller class. This makes it possible to use a constructor with a private final field as an argument without implementing it.

  3. The service

    used by the Controller is defined as a private final field. This field is injected into

    the Service instance when Quarkus starts.

  4. Set @Get to the method that is the endpoint for the API.

  5. Retrieves an Entity from Service, converts it to DTO, and returns it as a response.

Registration process

The processing sequence for registration of the registration screen is as follows.

Figure 2. Processing sequence during registration

  1. The user performs the registration operation.

  2. Page calls the Web API.

  3. ControlELR converts the posted DTO into an entity.

  4. The Controller invokes Service.

  5. Service calls the Repository.

  6. The repository will insert or update according to the entity's state.

  7. Repository returns the entity.

  8. Service returns the entity.

  9. The controller returns the entity's ID and version properties.
