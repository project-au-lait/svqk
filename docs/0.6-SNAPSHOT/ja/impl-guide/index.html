本書は、本プロジェクトにおける実装作業を効率的かつ一貫性を持って進めるための指針を提供することを目的としています。

本書は、本プロジェクトに参画する開発者を対象としています。開発者が本書に示されるプロセス、及びアーキテクチャ仕様書に示される標準を遵守することにより、高品質なシステムを効率的に開発することを目指します。

ガイドの読み進め方

本書は以下の順序で読み進めることを推奨します。 この順序は、開発者が実際に動くアプリケーションとソースコードを手元で参照しながら、本書、及びアーキテクチャ仕様書の要点を確認できるよう構成されています。

  1. 環境構築
    開発環境構築手順に従い、開発に使用するPC上に実行・開発環境を構築します。

  2. アプリケーションの確認
    構築した環境でアプリケーションを実行します。 アーキテクチャ仕様書のユースケースと照らし合わせてアプリケーションを操作し、実装されている機能を確認します。

  3. アプリケーションアーキテクチャの確認

    1. 論理的な構成
      アーキテクチャ仕様書のアプリケーション構成、コンポーネント構成を参照し、アプリケーションを構成するコンポーネントとその責務を確認します。

    2. 物理的な実装
      実装プロセスとそこで説明されている各コンポーネントのソースコードを参照し、コンポーネントの実装方法を確認します。

開発環境構築手順

必要なソフトウェアのインストール

開発には以下のソフトウェアが必要です。

  • Docker Desktop

  • JDK v21

  • Node.js v20

  • pnpm

  • Git

  • Visual Studio Code

次のOSごとの手順に従ってソフトウェアをインストールしてください。

Windows

WindowsではChocolateyを使用してソフトウェアをインストールします。

管理者権限でPowershellを起動します。(起動操作は下記参照)

  1. ショートカットキー Windows+R を実行します。

  2. 表示された ファイル名を選択して実行 ダイアログで powershell と入力します。

  3. ショートカットキー Ctrl+Shift+Enter を実行します。

表示されたPowershellウィンドウで以下のコマンドを実行します。

Set-ExecutionPolicy Bypass -Scope Process -Force; [System.Net.ServicePointManager]::SecurityProtocol = [System.Net.ServicePointManager]::SecurityProtocol -bor 3072; iex ((New-Object System.Net.WebClient).DownloadString('https://community.chocolatey.org/install.ps1'))

続けて以下のコマンドを実行し、必要なソフトウェアをインストールします。

choco install -y docker-desktop
choco install -y corretto21jdk
choco install -y nodejs-lts
choco install -y pnpm
choco install -y git
choco install -y vscode

macOS

macOSではHomebrewを使用してソフトウェアをインストールします。

ターミナルで以下のコマンドを実行し、Homebrewをインストールします。

/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

続けて以下のコマンドを実行し、必要なソフトウェアをインストールします。

brew install --cask docker
brew install --cask corretto@21
brew install node@20
brew install pnpm
brew install git
brew install --cask visual-studio-code

プロジェクト資材の取得とセットアップ

コマンドプロンプト(Windows)/ターミナル(macOS)で以下のコマンドを実行し、プロジェクト資材の取得と開発環境のセットアップを行います。

# プロジェクト資材の取得
git clone https://github.com/project-au-lait/svqk.git
cd svqk

# 以下、開発環境のセットアップ

## コマンドプロンプト(Windows)
chcp 65001
mvnw install -T 1C -P setup,browse-e2etest


## ターミナル(macOS)
./mvnw install -T 1C -P setup,browse-e2etest

上記のセットアップコマンドの処理内容、及び実現方式はビルド構成を参照してください。

プロジェクトを開く

以下のコマンドを実行し、プロジェクトをVSCodeワークスペースとして開きます。

code svqk.code-workspace

VSCodeワークスペースが開いたら、ウィンドウ右下に表示される以下のダイアログで Install ボタンをクリックして拡張機能をインストールします。

実装プロセス

実装プロセスは、ソースコードの実装方法とそのファイルの配置場所、及びそれらを実装する順序をガイドしたものです。

実装方法は、リファレンス実装を例にソースコードの各行の役割を説明していて、開発者はこれを参考に自身の担当範囲を実装することでアーキテクチャ仕様に準拠したソースコードが実装できます。

実装順序は、実装した結果をすぐにアプリケーションとして動かして確認できるよう構成されています。

実装プロセスは処理ごとに分類されていて、開発者は自身の担当範囲と同じ分類のプロセスを参照することができます。

登録画面

ここでは登録画面の実装プロセスを説明します。 説明ではリファレンス実装のチケット登録画面を例に挙げます。

Figure 1. 登録画面の例　チケット登録画面

登録画面には、登録内容を入力する項目と、登録を実行するボタンが配置されます。 また、登録画面では以下の2つの処理があります。

  1. 画面に遷移して来た際の初期化処理

  2. 画面に入力されたデータの登録処理

以降では、それぞれの処理に対する実装を説明します。

初期化処理

登録画面の初期化処理のシーケンスは以下の通りです。

Table 3. 画面初期表示時の処理シーケンス

  1. ユーザーが登録画面に遷移します。

  2. PageLoaderのload functionが呼び出されます。

  3. PageLoaderが画面の入力値を格納するmodelのインスタンスを生成します。インスタンスのプロパティには画面項目の初期値を設定します。

  4. PageLoaderが画面で使用するコード値等を取得するWeb APIを呼び出します。取得した結果はmodelとしてPageに返却します。

  5. ControllerがServiceを呼び出します。

  6. ServiceがRepositoryを呼び出します。

  7. RepositoryがDBに対しSELECTを実行します。

  8. ControllerがServiceから取得したentityをdtoに変換します。

  9. PageがPageLoaderから取得したmodelを元に画面のHTMLを構築します。また入力項目のバリデーションを定義します。

以降では、上記の処理シーケンスの各要素を以下の順に実装します。

  1. Frontend: 体裁とバリデーションの実装

  2. DB: テーブルを作成するマイグレーションスクリプトの実装

  3. Backend: Web APIの実装

  4. Frontend: Web API Clientの生成と組み込み

Frontend

登録画面の実装プロセスでは、初めにFrontendの体裁部分を実装します。

Page

Pageに画面項目を配置し、バリデーションの定義を実装します。 Pageのスクリプトセクションには、バリデーションの定義とマークアップセクションで必要な変数を宣言します。

svqk-front/src/routes/issues/[issueId]/+page.svelte スクリプトセクション
  import { IssueModel } from '$lib/arch/api/Api';
  import InputField from '$lib/components/InputField.svelte';
  import * as yup from 'yup';
  import FormValidator from '$lib/arch/form/FormValidator';

  const issue : IssueModel = {}; (1)

  const spec = {
    subject: yup.string().required()
  }; (2)

  const form = FormValidator.createForm(spec, () => {}); (3)

  1. Modelのインスタンスを生成します。この時点では仮実装で空のオブジェクトを生成します。

  2. yup.object 関数を使用し、バリデーションの仕様を定義します。仕様は、入力項目ごとのプロパティと、プロパティのバリデーションを定義したオブジェクトです。ここでは、 subject という名前のstring型の必須入力項目を定義しています。

  3. FormValidator.createForm 関数を使用し、バリデーションをhtml要素に適用するためのオブジェクトを生成します。この時点では第2引数は仮実装で空の関数「() ⇒ {}」を指定します。

Pageのマークアップセクションには画面項目を配置します。

svqk-front/src/routes/issues/[issueId]/+page.svelte マークアップセクション
<form use:form>  (1)
  <div>
    <InputField id="subject" label="題名" bind:value={issue.subject} />  (2)
  </div>


  <div>
    <button>登録</button> (3)
  </div>
</form>

  1. formタグを配置します。formタグにはスクリプトセクションで生成したformオブジェクトを設定します。

  2. divタグで1行分の領域を確保し、その中に入力項目を配置します。

  3. 処理を実行するボタンを配置します。

Frontendの実装は、Frontendサーバーを起動した状態で行います。 これにより、ブラウザで画面の体裁とバリデーションの挙動を確認しながら実装を進めることができます。 Frontendサーバーの起動方法はプロジェクトの使い方を参照してください。

Backend

BackendにDBから値を取得するWeb APIを実装します。

Entity

Entityを生成するための設定をjeg設定ファイル(jeg-config.yml)に追加します。

svqk-entity/src/tool/resources/jeg-config.yml

  1. packages属性にEntityの生成先パッケージとテーブル名を追加します。

    • "Entityの生成先パッケージ" : ["テーブル名"]

jeg設定ファイルを更新したらVSCode Task: gen-entity を実行し、entityプロジェクト以下にEntityのjavaファイルが生成されていることを確認します。

Repository

Repositoryのjavaファイルをドメインパッケージ以下に作成します。

svqk-back/src/main/java/dev/aulait/svqk/domain/issue/IssueStatusRepository.java
package dev.aulait.svqk.domain.issue;

import org.springframework.data.jpa.repository.JpaRepository;

public interface IssueStatusRepository extends JpaRepository<IssueStatusEntity, String> { (1)
}

  1. JpaRepositoryを継承します。型パラメータにはEntityとEntityのIDの型を指定します。

Service

Serviceのjavaファイルをドメインパッケージ以下に作成します。

svqk-back/src/main/java/dev/aulait/svqk/domain/issue/IssueService.java
package dev.aulait.svqk.domain.issue;

import java.util.List;

import jakarta.enterprise.context.ApplicationScoped;
import lombok.RequiredArgsConstructor;

@ApplicationScoped (1)
@RequiredArgsConstructor (2)
public class IssueStatusService {

  private final IssueStatusRepository statusRepository; (3)

  public List<IssueStatusEntity> findAll() {
    return statusRepository.findAll(); (4)
  }

}

  1. @ApplicationScopedをServiceクラスに設定します。

  2. @RequiredArgsConstructorをServiceクラスに設定します。これにより、private finalフィールドを引数に持つコンストラクタが実装しなくても使用可能になります。

  3. Serviceが使用するRepositoryをprivate finalフィールドとして定義します。このフィールドはQuarkus起動時にRepositoryのインスタンスがInjectされます。

  4. RepositoryからEntityを取得します。

DTO

DTOのjavaファイルをインターフェースパッケージ以下に作成します。

package dev.aulait.svqk.interfaces.issue;

import org.eclipse.microprofile.openapi.annotations.media.Schema;

import lombok.Data;

@Data (1)
public class IssueStatusDto {

  (2)
  @Schema(required = true, readOnly = true)

Controller

Controllerのjavaファイルをインターフェースパッケージ以下に作成します。

svqk-back/src/main/java/dev/aulait/svqk/interfaces/issue/IssueStatusController.java
package dev.aulait.svqk.interfaces.issue;

import java.util.List;

import dev.aulait.svqk.arch.util.BeanUtils;
import dev.aulait.svqk.arch.web.ApiPath;
import dev.aulait.svqk.domain.issue.IssueStatusService;
import jakarta.ws.rs.GET;
import jakarta.ws.rs.Path;
import lombok.RequiredArgsConstructor;

@Path(IssueStatusController.ISSUE_STATUSES_PATH) (1)
@RequiredArgsConstructor (2)
public class IssueStatusController {

  private final IssueStatusService service; (3)

  static final String ISSUE_STATUSES_PATH = ApiPath.ROOT + "/issue-statuses";

  @GET (4)
  public List<IssueStatusDto> get() {
    return BeanUtils.mapAll(service.findAll(), IssueStatusDto.class); (5)
  }
}

  1. @PathをControllerクラスに設定します。これにより、このControllerクラスがAPIのエンドポイントして認識されます。

  2. @RequiredArgsConstructorをControllerクラスに設定します。これにより、private finalフィールドを引数に持つコンストラクタが実装しなくても使用可能になります。

  3. Controllerが使用するServiceをprivate finalフィールドとして定義します。このフィールドはQuarkus起動時にServiceのインスタンスがInjectされます。

  4. @GetをAPIのエンドポイントとなるメソッドに設定します。

  5. ServiceからEntityを取得し、それをDTOに変換したものをレスポンスとして返却します。

Backendの実装が終わったらBackendサーバーを起動し、Swagger UIを使用して動作確認を行います。 Backendサーバーの起動方法はプロジェクトの使い方を、Swagger UIのアクセス先はアクセス情報をそれぞれ参照してください。
Frontend

API Clientを生成し、Web APIの呼び出し処理を追加します。

API Client

VSCode Task: gen-api-client を実行し、API Clientを生成します。 生成されたAPI Clientは svqk-front/src/lib/arch/api/Api.ts ファイルに出力されます。

PageLoader
svqk-front/src/routes/issues/[issueId]/+page.ts
import type { IssueModel, IssueStatusModel } from '$lib/arch/api/Api';
import ApiHandler from '$lib/arch/api/ApiHandler';
import type { PageLoad } from './$types';

export const load: PageLoad = async ({ fetch, params }) => {  (1)

  const issueStatuses = (await ApiHandler.handle<IssueStatusModel[]>(fetch, (api) =>
    api.issueStatuses.issueStatusesList()
  ))!; (2)

  return {
    issueStatuses (3)
  };
};

  1. load関数を定義します。

  2. ApiHandler.handle関数を使用してWeb APIを呼び出します。

  3. Web APIの呼び出し結果を返却します。

  export let data: PageData;  (1)

  1. PageLoaderからの返却値を格納する変数を宣言します。

  2. 返却値を変数に格納します。

Frontendサーバー、Backendサーバー、DBを全て起動した状態で画面操作を行い、期待通りの挙動になること、BackendサーバーのログにWeb APIの処理のログ等が出力されることを確認します。

登録処理

登録画面の登録時の処理シーケンスは以下の通りです。

Figure 2. 登録時の処理シーケンス

  1. ユーザーが登録操作を実行します。

  2. PageがWeb APIを呼び出します。

  3. ControlelrがPOSTされたdtoをentityに変換します。

  4. ControllerがServiceを呼び出します。

  5. ServiceがRepositoryを呼び出します。

  6. Repositoryがentityの状態に応じてINSERTまたはUPDATEを実行します。

  7. Repositoryがentityを返却します。

  8. Serviceがentityを返却します。

  9. Controllerがentityのidとversionプロパティを返却します。
