本書の対象読者は、本システムの開発・保守における実装、自動テスト、CI・CD構築を担当する開発者です。 本書の記載内容は、本システム、およびその開発・保守における、アプリケーション、開発・CI・CD環境、自動テストの側面に対するものです。 また、これら側面において開発者が作成する成果物、主にアプリケーションのソースコード、テストスクリプト、ビルドスクリプト、設定ファイルを作成する際の規則、即ち実装方式を定義します。 規則化の対象となるのは、ソースコードをファイルに記載する際の、記載内容の複数ファイルへの分割、ファイル内でのプログラミング言語の構文に則った構成要素への分割、ファイルの配置ディレクトリ、それらの命名等です。本書では、このソースコードの記載内容の分割において、同種の記載内容の集合を責務と呼び、同種の責務を割り当てられた集合をコンポーネントと呼びます。 コンポーネントの構成、及びそれらの相互作用を可視化することで、開発者による規則の理解を助けることを目指します。

アプリケーション

ここでは、本システムにおいて開発対象となるアプリケーションの実装方式を定義します。 まず、アプリケーションとそれらの責務と相互作用を定義します。 次に、アプリケーションをコンポーネントに分割し、それらの責務と相互作用定義します。 最後に、各コンポーネントの実装と配置の規則を定義します。

アプリケーション構造

本システムは、FrontendとBackendの2つのアプリケーションとDBで構成されます。

Figure 1. アプリケーション構造

FrontendはSvelteKitで実装されたWebアプリケーションです。 ユーザーに対しブラウザで利用可能なユーザーインターフェースを提供します。 また、ユーザーの入力に対する簡易的なバリデーションや、Web APIの呼び出しを行います。

BackendはQuarkusで実装されたWeb APIアプリケーションです。 フロントエンドアプリケーションに対しWeb APIを提供します。 Web API内部では、リクエストに対する厳密なバリデーションや、DBのトランザクション制御、ビジネスロジック、DBアクセスを行います。

Page

Pageは、画面項目の配置、バリデーションの定義、画面イベントのハンドリング、Web APIの呼び出しを担うコンポーネントです。

Pageは、SvelteKitのRoutingで使用する +page.svelte ファイルで実装します。+page.svelte ファイルとRoutingとの関係は以下の通りです。

  • +page.svelte ファイルはsrc/routes以下の任意の階層に配置します。

  • +page.svelte ファイルを配置したsrc/routes以下のパスが、当該画面にアクセスするURLのパスになります。

    • 例
      配置: src/routes/a/b/c/+page.svelte
      URL: http://host:port/a/b/c

SvelteKitのRoutingと +page.svelte ファイルの詳細な仕様はSvelteKitのドキュメントを参照してください。

  • SvelteKit Routing

  • SvelteKit Routing +page.svelte

Pageの実装内容は以下の通りです。

+page.svelte
<script lang="ts">

  // script section (1)

</script>

 (2)
 <!-- markup section -->
1 スクリプトセクションには、TypeScriptでバリデーションの定義や画面イベントのハンドリング、Web APIを呼び出し等を実装します。
2 マークアップセクションには、HTML + Svelteの構文で画面項目の配置を実装します。

PageLoader

PageLoaderは、SvelteKitのLoading dataで使用する +page.ts ファイルで実装します。

  • +page.ts ファイルは画面の +page.svelte ファイルと同じディレクトリに配置する。

Controller

Controllerは、Web APIのエンドポイントとなるコンポーネントです。クライアントからHTTPリクエストを受け取り、それに対するHTTPレスポンスを返します。Validator、Factory、Serviceの実行順序と入出力を制御することで、Web API内部の処理フローを制御します。 Controllerの実装には、Jakarta RESTを使用します。

Controllerの実装
package dev.aulait.svqk.interfaces.hello;

import dev.aulait.svqk.domain.hello.HelloEntity;
import dev.aulait.svqk.domain.hello.HelloService;
import jakarta.validation.Valid;
import jakarta.ws.rs.GET;
import jakarta.ws.rs.POST;
import jakarta.ws.rs.Path;
import jakarta.ws.rs.PathParam;
import lombok.RequiredArgsConstructor;

@Path(HelloController.HELLO_PATH) (1)
@RequiredArgsConstructor (2)
public class HelloController {

  private final HelloService helloService; (3)

  static final String HELLO_PATH = "hello";

  static final String HELLO_GET_PATH = "{id}";

  @GET
  @Path(HELLO_GET_PATH)
  public HelloDto get(@PathParam("id") int id) { (4)

    HelloEntity entity = helloService.find(id);

    return HelloDto.builder().id(entity.getId()).message(entity.getMessage()).build();
  }

  @POST
  public int save(@Valid HelloDto dto) { (4)

    HelloEntity entity = HelloEntity.builder().id(dto.getId()).message(dto.getMessage()).build();

    HelloEntity savedEntity = helloService.save(entity);

    return savedEntity.getId();
  }
}
1 @Path をControllerクラスに設定します。これにより、このControllerクラスがWeb APIのエンドポイントしてQuarkusに認識されます。
2 @RequiredArgsConstructor をControllerクラスに設定します。これにより、private finalフィールドを引数に持つコンストラクタがコンパイル時に自動生成されます。
3 Controllerが使用するコンポーネントをprivate finalフィールドとして定義します。このフィールドは、Quarkus起動時に該当するコンポーネントのインスタンスがInjectされます。
4 Web APIのエンドポイントとなるメソッドを定義します。メソッドにはWeb APIのHTTP Methodを表すアノテーション(@GET、@POST 等)を設定します。

Controllerの実装で使用可能なJakarta RESTの機能の詳細はQuarkusのガイドを参照してください。

  • Writing REST Services with Quarkus REST

Service

Serviceは、トランザクション処理を担うコンポーネントです。Repository、Logicの実行順序と入出力を制御することでトランザクション処理のフローを制御します。 Serviceの実装には、JTA、CDIを使用します。

Serviceの実装
package dev.aulait.svqk.domain.hello;

import jakarta.enterprise.context.ApplicationScoped;
import jakarta.transaction.Transactional;
import lombok.RequiredArgsConstructor;

@ApplicationScoped (1)
@RequiredArgsConstructor (2)
public class HelloService {

  private final HelloRepository helloRepository; (3)

  public HelloEntity find(int id) { (4)

  }

  @Transactional
  public HelloEntity save(HelloEntity entity) { (4)

  }
}
1 @ApplicationScoped をServiceクラスに設定します。これにより、ServiceクラスのインスタンスがCDI Managed Beanとして使用可能になります。
2 @RequiredArgsConstructor をServiceクラスに設定します。これにより、private finalフィールドを引数に持つコンストラクタが実装しなくても使用可能になります。
3 Serviceが使用するコンポーネントをprivate finalフィールドとして定義します。このフィールドはQuarkus起動時に該当するコンポーネントのインスタンスがInjectされます。
4 トランザクション処理ごとにメソッドを定義します。DBへの書き込みを行う処理のメソッドには @Transactinal を設定します。

Quarkusで使用可能なJTAの機能の詳細はQuarkusのドキュメントを参照してください。

DTO

DTOは、Web APIのリクエスト及びレスポンスを表すオブジェクトです。

DTOの実装
package dev.aulait.svqk.interfaces.hello;

import lombok.Builder;
import lombok.Data;
import org.eclipse.microprofile.openapi.annotations.media.Schema;

@Data (1)
@Builder
public class HelloDto {
  @Schema(required = true) (2)
  private Integer id;

  private String message;
}
1 @Data をDTOクラスに設定します。これにより、フィールドのGetter、Setter等が自動生成されます。
2 リクエスト及びレスポンスのプロパティとなるフィールドを定義します。

  • リクエスト及びレスポンスのJSONプロパティに必ず登場させるフィールドに対しては @Schema(required = true) を指定します。NOT NULLのカラムの値を格納するフィールド等がこれに該当します。
